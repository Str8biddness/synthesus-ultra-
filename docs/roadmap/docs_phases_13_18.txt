OVERVIEW

The Social Fabric system enables NPC-to-NPC relationships, faction dynamics, gossip propagation, dispositions, group conversations, and messaging — creating a living social world independent of player interaction.

File: cognitive/social_fabric.py (580 lines)
Tests: tests/test_social_fabric.py (95 tests)

ARCHITECTURE

Core Data Models:
  • NPCProfile — name, faction_ids, location, known_gossip, dispositions, message_inbox
  • Faction — faction_id, name, members, leader, description
  • FactionRelation — allied, neutral, hostile, rival (between faction pairs)
  • GossipItem — gossip_id, content, source_npc, subject, priority, truth_value, decay_per_hop, hops, heard_by, tags
  • GossipPriority — TRIVIAL, NORMAL, IMPORTANT, URGENT
  • NPCMessage — sender_id, recipient_id, content, message_type (chat/warning/request/gossip/trade)
  • GroupConversation — group_id, participants (with roles), messages, topic, max_messages
  • ConversationRole — INITIATOR, PARTICIPANT, OBSERVER

Key APIs:
  register_npc(npc_id, name, faction_ids, location) — Register NPC in the fabric
  create_faction(name, faction_id) — Create a faction
  set_faction_relation(faction_a, faction_b, relation) — Set relation between factions
  set_disposition(from_npc, to_npc, value) — Set how one NPC feels about another (-1 to 1)
  get_disposition(from_npc, to_npc) — Get NPC-to-NPC feeling
  create_gossip(source_npc, content, priority, ...) — Create gossip that propagates
  send_message(sender_id, recipient_id, content, msg_type) — Direct NPC-to-NPC messaging
  start_group_conversation(initiator_id, participant_ids, topic) — Start group chat
  end_group_conversation(group_id) — End group conversation
  tick() — World tick: propagate gossip, process interactions

GOSSIP PROPAGATION
  • Gossip spreads during tick() to NPCs in the same location or faction
  • Each hop reduces truth_value by decay_per_hop
  • Priority affects propagation speed (URGENT spreads faster)
  • NPCs track which gossip they've heard via known_gossip (set of IDs)
  • Gossip can have tags for topic filtering

DISPOSITION SYSTEM
  • Float value from -1.0 (hostile) to 1.0 (adoring)
  • Faction relations provide base disposition between faction members
  • Individual disposition overrides faction defaults
  • Dispositions affect gossip sharing willingness

COST
  • Tick: <0.2ms for 100 NPCs
  • Memory: ~2KB per NPC profile
  • Zero GPU

OVERVIEW

A C++ kernel bridge providing native-speed pattern matching via pybind11, with IPC and Python fallback modes. Enables hot-path acceleration for game engines.

Files:
  kernel/bridge.py (516 lines) — Python bridge API + fallback mode
  kernel/pybind_module.cpp (181 lines) — pybind11 native module
  CMakeLists.txt (152 lines) — Full CMake build system
Tests: tests/test_kernel_bridge.py (45 tests)

ARCHITECTURE

Three operating modes (auto-detected):
  1. NATIVE — pybind11 compiled module (fastest, requires C++ build)
  2. IPC — Subprocess communication (medium speed, cross-process)
  3. FALLBACK — Pure Python implementation (always available)

KernelBridge API:
  • query(KernelQuery) → KernelResult — Route a query through PPBRS
  • mode — Current operating mode (BridgeMode enum)
  • shutdown() — Clean up resources

KernelQuery: text, context, priority
KernelResult: response, confidence, module_used, latency_ms, metadata

PPBRS (Pattern-Priority Batch Routing System):
  • add_route(keywords, route_name) — Register keyword→route mapping
  • query(text) → (route, confidence) — Route based on keyword overlap
  • Token-based matching with confidence scoring

C++ Module (pybind11):
  • synthesus_kernel.PatternMatcher — Native pattern matching
  • synthesus_kernel.PPBRS — Native routing
  • synthesus_kernel.version() — Version info

CMake Build:
  • Requires: pybind11, C++17
  • Builds: synthesus_kernel.so shared library
  • Install target copies to kernel/ directory

PERFORMANCE
  • Fallback mode: 0.006ms average per query
  • Native mode (estimated): 0.001ms per query
  • Throughput: ~170,000+ queries/sec (fallback), ~1M+ (native)

Tests: tests/test_sdk.py (21 tests)

PYTHON SDK

SynthesusClient(host, port, api_key):
  • query(npc_id, player_id, text) → NPCResponse
  • get_character(npc_id) → CharacterInfo
  • list_characters() → List[CharacterInfo]
  • set_world_state(flags) — Update world flags
  • get_world_state() → WorldState
  • health() → HealthStatus
  • batch_query(queries) → List[NPCResponse] — Parallel queries
  • stream_query(npc_id, player_id, text) → AsyncIterator — Streaming responses

NPCResponse: text, confidence, emotion, source, relationship, debug
CharacterInfo: character_id, name, role, pattern_count, has_knowledge, has_personality
WorldState: flags (dict), timestamp
HealthStatus: status, uptime, active_npcs, version

UNITY SDK (C#)
  • SynthesusClient MonoBehaviour component
  • Coroutine-based async queries
  • NPCResponse, CharacterInfo, WorldState models
  • Inspector-configurable host/port/API key
  • Automatic retry with exponential backoff

UNREAL SDK (C++)
  • USynthesusClient UObject component
  • FHttpModule-based async HTTP
  • FSynthesusNPCResponse, FSynthesusCharacterInfo structs
  • Blueprint-compatible (UFUNCTION, UPROPERTY macros)
  • Delegate-based callbacks
